# TLS Termination with NGINX and the azihsm OpenSSL Provider

Terminate TLS in NGINX using private keys that never leave the Azure Integrated HSM. NGINX loads keys at runtime through the OpenSSL 3.x store API — no patches or custom modules required.

## How It Works

NGINX mainline (>= 1.27) supports the OpenSSL `OSSL_STORE` API. When `ssl_certificate_key` contains a `store:` URI, NGINX delegates key loading to the registered store provider. The azihsm provider handles `azihsm://` URIs by reading a masked key blob from disk, unmasking it inside the HSM, and returning an `EVP_PKEY` backed by HSM key handles.

```
                  OPENSSL_CONF
                       │
                       ▼
┌─────────┐    ┌───────────────┐    ┌─────────┐
│  NGINX  │───>│azihsm provider│───>│   HSM   │
└─────────┘    └───────────────┘    └─────────┘
     │                 │
     │  ssl_certificate_key         masked_key.bin
     │  "store:azihsm://..."        (opaque blob on disk)
```

The provider is loaded via `OPENSSL_CONF` — NGINX itself is unmodified.

## Prerequisites

- **NGINX mainline >= 1.27** — required for `OSSL_STORE` / `store:` URI support in `ssl_certificate_key`. The stable branch does not support this.
- **OpenSSL 3.0.3** — the azihsm provider is built and tested against this version.
- **azihsm provider** (`azihsm_provider.so`) and its runtime dependency (`libazihsm_api_native.so`) installed system-wide. See the [provider README](../README.md#installation) for build and installation steps.

## Step 1 — Install NGINX Mainline

The stable branch does not support `store:` URIs. Install mainline from the official repository:

```bash
curl -fsSL https://nginx.org/keys/nginx_signing.key \
  | sudo gpg --dearmor -o /usr/share/keyrings/nginx-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/nginx-archive-keyring.gpg] \
  https://nginx.org/packages/mainline/ubuntu noble nginx" \
  | sudo tee /etc/apt/sources.list.d/nginx.list

sudo apt-get update
sudo apt-get install -y nginx
```

## Step 2 — Prepare Key Material

The provider requires several key material files at known paths. Create a dedicated directory and generate the required files:

```bash
sudo mkdir -p /var/lib/azihsm
```

| File | Description | Notes |
|------|-------------|-------|
| `bmk.bin` | Backup Masking Key | Auto-generated by the provider on first use |
| `muk.bin` | Masked Unwrapping Key | Auto-generated by the provider on first use |
| `obk.bin` | Owner Backup Key (48 bytes) | Must be pre-generated |
| `pota_private_key.der` | POTA P-384 private key (EC DER) | Must be pre-generated |
| `pota_public_key.der` | POTA P-384 public key (SPKI DER) | Must be pre-generated |

```bash
OSSL=/opt/openssl-3.0.3/bin/openssl

# OBK — 48-byte random key
$OSSL rand -out /var/lib/azihsm/obk.bin 48

# POTA — P-384 key pair
$OSSL ecparam -name secp384r1 -genkey -noout \
  | $OSSL ec -outform DER -out /var/lib/azihsm/pota_private_key.der 2>/dev/null
$OSSL ec -in /var/lib/azihsm/pota_private_key.der -inform DER \
  -pubout -outform DER -out /var/lib/azihsm/pota_public_key.der 2>/dev/null

# Lock down permissions
sudo chmod 600 /var/lib/azihsm/*.bin /var/lib/azihsm/*.der
```

## Step 3 — Create the OpenSSL Provider Config

This config file tells OpenSSL to load the azihsm provider. NGINX reads it via the `OPENSSL_CONF` environment variable.

Save as `/etc/azihsm/openssl-provider.cnf`:

```ini
openssl_conf = openssl_init

[openssl_init]
providers = provider_sect

[provider_sect]
default = default_sect
base = base_sect
azihsm_provider = azihsm_provider_sect

[default_sect]
activate = 1

[base_sect]
activate = 1

[azihsm_provider_sect]
module = /usr/lib/x86_64-linux-gnu/ossl-modules/azihsm_provider.so
activate = 1
azihsm-bmk-path  = /var/lib/azihsm/bmk.bin
azihsm-muk-path  = /var/lib/azihsm/muk.bin
azihsm-obk-path  = /var/lib/azihsm/obk.bin
azihsm-mobk-path = /var/lib/azihsm/mobk.bin
azihsm-obk-source = caller
azihsm-pota-source = caller
azihsm-pota-private-key-path = /var/lib/azihsm/pota_private_key.der
azihsm-pota-public-key-path  = /var/lib/azihsm/pota_public_key.der
```

All key material paths must be absolute. If they are omitted, the provider falls back to CWD-relative defaults (`./bmk.bin`, etc.), which breaks when NGINX's working directory does not contain the files.

> The `base` provider is required — NGINX uses it for PEM decoding of `ssl_certificate` files.

> **TPM source:** When POTA and OBK are managed by the TPM (`azihsm-obk-source = tpm` and `azihsm-pota-source = tpm`), the `azihsm-obk-path`, `azihsm-pota-private-key-path`, and `azihsm-pota-public-key-path` settings are not required and can be omitted. The `obk.bin` and POTA key files from Step 2 are also not needed in this case.

## Step 4 — Generate a TLS Key and Certificate

Use explicit `-provider` flags to generate an EC P-384 key inside the HSM and a self-signed certificate. This approach avoids `OPENSSL_CONF` entirely for key generation, preventing the provider from being loaded twice (once from the config and once from the CLI flags).

```bash
OSSL=/opt/openssl-3.0.3/bin/openssl
PROV_PATH=/usr/lib/x86_64-linux-gnu/ossl-modules

export LD_LIBRARY_PATH=/opt/openssl-3.0.3/lib
export AZIHSM_CREDENTIALS_ID="<your-hex-credential-id>"
export AZIHSM_CREDENTIALS_PIN="<your-hex-credential-pin>"

# OPENSSL_CONF must be unset — explicit flags prevent double-loading
unset OPENSSL_CONF

# CWD must contain the key material (obk.bin, pota keys) because the
# provider falls back to CWD-relative defaults when not using a config.
cd /var/lib/azihsm

# Generate an EC P-384 key (writes a masked key blob to disk)
$OSSL genpkey \
  -provider-path $PROV_PATH \
  -provider default -provider azihsm_provider \
  -propquery "?provider=azihsm" \
  -algorithm EC \
  -pkeyopt group:P-384 \
  -pkeyopt "azihsm.masked_key:/etc/azihsm/masked_key_p384.bin" \
  -outform DER -out /dev/null

# Generate a self-signed certificate referencing the masked key
$OSSL req -new -x509 \
  -provider-path $PROV_PATH \
  -provider default -provider azihsm_provider \
  -propquery "?provider=azihsm" \
  -key "azihsm:///etc/azihsm/masked_key_p384.bin;type=ec" \
  -subj "/CN=your-hostname" \
  -days 365 -sha384 \
  -out /etc/azihsm/server.crt
```

The `-out /dev/null` on `genpkey` discards the standard OpenSSL key output. The actual output is the masked key blob at the path specified by `azihsm.masked_key`. This blob is opaque and only usable on the same HSM that created it.

The provider also auto-generates `bmk.bin` and `muk.bin` in the CWD during this step. These files are needed at runtime, so they must remain in `/var/lib/azihsm/`.

## Step 5 — Configure NGINX

The critical directive is `ssl_certificate_key` with a `store:azihsm://` URI. This tells NGINX to load the key through the provider's store implementation rather than reading a PEM file directly.

```nginx
worker_processes 1;

# Pass environment variables to the worker process.
# Without these, the provider cannot authenticate with the HSM.
env OPENSSL_CONF;
env AZIHSM_CREDENTIALS_ID;
env AZIHSM_CREDENTIALS_PIN;

error_log /var/log/nginx/error.log info;
pid       /run/nginx.pid;

events {
    worker_connections 1024;
}

http {
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_session_cache   shared:SSL:10m;
    ssl_session_tickets off;

    server {
        listen      443 ssl;
        server_name your-hostname;

        ssl_certificate     /etc/azihsm/server.crt;
        ssl_certificate_key "store:azihsm:///etc/azihsm/masked_key_p384.bin;type=ec";

        # Your location blocks here
        location / {
            proxy_pass http://backend;
        }
    }
}
```

The `env` directives are required at the top level (before `events`). NGINX strips the environment by default when spawning worker processes. Without these directives, `OPENSSL_CONF` and the credential variables are not visible to the provider.

## Step 6 — Start NGINX

Use `sudo env` with explicit environment variables instead of `sudo -E`. This prevents `LD_LIBRARY_PATH` from leaking into the NGINX process (NGINX links against the system OpenSSL, not the custom 3.0.3 build) and bypasses restrictive `sudoers` `env_reset` policies:

```bash
# Validate the configuration (tests that the provider loads and the key is readable)
sudo env -u LD_LIBRARY_PATH \
  OPENSSL_CONF=/etc/azihsm/openssl-provider.cnf \
  AZIHSM_CREDENTIALS_ID="<your-hex-credential-id>" \
  AZIHSM_CREDENTIALS_PIN="<your-hex-credential-pin>" \
  nginx -t

# Start NGINX
sudo env -u LD_LIBRARY_PATH \
  OPENSSL_CONF=/etc/azihsm/openssl-provider.cnf \
  AZIHSM_CREDENTIALS_ID="<your-hex-credential-id>" \
  AZIHSM_CREDENTIALS_PIN="<your-hex-credential-pin>" \
  nginx

# Verify TLS is working
curl -k https://localhost/
```

**Why `env -u LD_LIBRARY_PATH`?** The custom OpenSSL 3.0.3 build is only needed by the `openssl` CLI commands in Step 4. NGINX links against the system OpenSSL at runtime. If `LD_LIBRARY_PATH` leaks through, NGINX may pick up the wrong `libcrypto.so.3` and crash or behave unexpectedly.

## Verifying the Certificate

Confirm the TLS certificate uses the expected algorithm and curve:

```bash
echo | openssl s_client -connect localhost:443 -servername your-hostname 2>/dev/null \
  | openssl x509 -noout -text
```

Look for:
- `Signature Algorithm: ecdsa-with-SHA384`
- `ASN1 OID: secp384r1` / `NIST CURVE: P-384`
- `Subject: CN = your-hostname`

## Troubleshooting

### "unregistered scheme" when running `nginx -t`

The provider is not loaded. Check:
- `OPENSSL_CONF` is set and points to `openssl-provider.cnf`
- The `module` path in the config matches where `azihsm_provider.so` is installed
- `libazihsm_api_native.so` is in the library path (`ldconfig -p | grep azihsm`)
- NGINX was started with `sudo env OPENSSL_CONF=...` (not plain `sudo`)

### Credential or authentication errors

The HSM credentials are not reaching the worker. Ensure:
- `env OPENSSL_CONF;`, `env AZIHSM_CREDENTIALS_ID;`, and `env AZIHSM_CREDENTIALS_PIN;` are present in `nginx.conf` at the top level
- The environment variables are passed explicitly in the `sudo env` command
- `LD_LIBRARY_PATH` is stripped with `-u LD_LIBRARY_PATH` to avoid loading the wrong OpenSSL

### Key loading fails silently

- Verify key material files exist at the paths in `openssl-provider.cnf` and have correct permissions
- Check `/var/log/nginx/error.log` for provider error messages
- Run `nginx -t` first — it validates the full config including key loading

### NGINX stable does not recognize `store:` URIs

The `ssl_certificate_key "store:..."` syntax requires **NGINX mainline >= 1.27**. The stable branch does not support `OSSL_STORE`. Verify your version with `nginx -v`.

### Certificate generated but NGINX cannot read it

The `base` provider must be loaded alongside `azihsm_provider` in `openssl-provider.cnf`. NGINX uses the base provider for PEM decoding of `ssl_certificate`. If the base provider section is missing, NGINX cannot parse the certificate file.
